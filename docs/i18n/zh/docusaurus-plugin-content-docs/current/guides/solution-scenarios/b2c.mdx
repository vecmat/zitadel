## 对消费者的业务

一般用户有不同的需要。 您可能有最终用户、员工，甚至其他组队的客户 (B2B)。 这组用户通常不会有相同的意图，也不会以不同的方式使用应用。 当您规划应用程序时。 投资时间研究您的应用架构对于以后的功能升级和增强是至关重要的，因为如果您必须做出更大的更改，这些更改是与肝价格点联系在一起的。

本指南向您介绍了ZITADEL项目的分组和结构，这些项目构成了所有项目的基础。 这可以用来快速启动 [B2B 场景](./b2b)， 如果你有多个组织的项目，它仅仅侧重于规划考虑。

本指南的旅程始于创建一个组织，一个您身上最外层的ZITADEL层。 它是供项目、作用、应用程序和用户使用的船只。 可以从 [ZITADEL控制台](https://{your_domain}.zitadel.cloud/ui/console/orgs/create) 创建。 您可以为机构所有者选择您的当前帐户或创建一个新帐户。

根据您的软件开发生命周期(SDLC)，您可以创建多个组织或项目来保持应用程序环境的分离。

### 自定义域

在 org 创建之后，您将会收到您组织的域部分的问候。 ZITADEL自动创建窗体 `[orgname].zitadel.cloud`的自定义域名，但您可以创建自己的域名。 根据在 [域策略](../../concepts/structure/policies) 上的设置，必须通过在指定位置保存验证文件来验证它。 我们建议您尽早创建您的域名，因为这些域名为您的应用程序创造信任感，稍后的更改可能会产生额外的迁移努力。 您可以在这里阅读更多关于ZITADEL如何处理用户名 [](../manage/console/organizations#how-zitadel-handles-usernames)

### 数据提供

ZITADEL为您提供了一个基本的用户存储，并管理电话和电子邮件地址。 它还允许您存储您自己的应用程序数据，例如偏好设置或外部标识符到用户的元数据。

如果您正在迁移一个现有的项目，并且您已经有一个外部身份商店，您可以考虑批量导入您的用户数据集。 阅读我们的 [管理 API 定义](../../apis/proto/management#importhumanuser) 以获取更多信息。 如果用户电子邮件未验证或没有设置密码，将发送初始化邮件。

:::info
对管理 API 的请求是受限制的。 阅读我们的 [率限制策略](../../legal/rate-limit-policy) 以获取更多信息。
:::

### 用户认证

用户认证可以通过多种方式进行。 ZITADEL中的默认方法是启用MFA 的用户名和密码。 ZITADEL允许您配置多因子和无密码认证，以加强用户的安全。 所有身份验证方法都可以从免费等级上获得。 要设置您的组织登录政策，请前往 [控制台](https://{your_domain}.zitadel.cloud/ui/console/org) 中的组织细节。

在计划应用程序时考虑以下关于用户身份验证的问题：

- 什么方法来验证您的用户？
- 您的用户将在哪里输入凭据？
- 您是否提供密码验证？
- 您做了什么来保证您的用户凭据安全？
- 您是否提供多重身份验证？
- 您是否通过身份提供商登录？
- 您必须提供哪种语言？

在审视这个问题时， 你可能必须承认，建立身份管理系统比你最初想象的要复杂得多，如果你自己可能做得太多的话，就要实施。 特别是因为你应该聚焦于构建你的应用程序。

### Federation

ZITADEL支持与IDC 身份提供商以及JWT 身份提供商进行登记。 注册时，ZITADEL会将用户信息导入到自己的个人资料中。 //TODO 扩展此段

### 托管登录

ZITADEL提供一个“默认方法安全”并带有托管登录，它是您的用户进行身份验证的固定端点。 它是安全可靠的，并具有多因素、Feder登录和无密码功能。 OIDC (OpenID Connect) 打开门打开单点登录 (SSO)。 特别是如果您有多个应用程序，您可能需要一个中心和熟悉的认证体验。 通过SSO, ZITADEL提供了您所有应用的无缝体验。

### 项目和应用

由于我们托管的登录是一个单独的身份验证屏幕，您必须确定您如何将用户引向您的应用程序。 ZITADEL的申请生活在ZITAL项目之下。 您可以为您的不同客户端类型添加多个应用程序(Nant、Web、用户代理或 API)。 设置您的应用程序时考虑阅读我们关于 [身份验证流动](../integrate/login-users) 的指南。

### 访问控制

通过验证用户，您需要确保用户和服务能够访问您的应用程序和API。 这个过程称为访问控制，包含用户身份验证、授权和政策执行。 考虑到以下因素：

- 您的应用程序是否打电话给您自己的API？
- 您的应用程序是否需要调用第三方API？
- **您真的** 为第三方应用程序提供 API 吗？

检查用户是否有权访问特定的 API 所需的数据存储在用户授予范围内。 此信息通常存储在角色或自定义声明中，可以使用 `访问` 或 OIDC `id` 令牌访问。

在我们的 [指南](../integrate/oauth-recommended-flows) 中阅读更多关于授权的信息。
